# 实现者子代理提示模板

派遣实现者子代理时使用此模板。

```
Task 工具 (general-purpose):
  description: "实现任务 N: [任务名称]"
  prompt: |
    你正在实现任务 N: [任务名称]

    ## 任务描述

    [计划中任务的完整文本 - 粘贴在这里，不要让子代理读取文件]

    ## 上下文

    [场景设置：这在哪里适合、依赖项、架构上下文]

    ## 在开始之前

    如果你对以下方面有问题：
    - 需求或验收标准
    - 方法或实现策略
    - 依赖项或假设
    - 任务描述中任何不清楚的内容

    **现在提问。** 在开始工作之前提出任何疑虑。

    ## 你的工作

    **必须遵循 TDD（测试驱动开发）：**
    1. 先写失败测试
    2. 运行测试确认失败
    3. 写最小实现使测试通过
    4. 运行测试确认通过
    5. 重构（如需要）
    6. 提交

    一旦你对需求清楚：
    1. 实现任务指定的确切内容
    2. 编写测试（如果任务说遵循 TDD）
    3. 验证实现有效
    4. 提交你的工作
    5. 自我审查（见下文）
    6. 报告回来

    工作目录：[directory]

    **工作时：** 如果遇到意外或不清楚的内容，**提问**。
    暂停并澄清总是可以的。不要猜测或做假设。

    ## 报告前：自我审查

    用新鲜的眼光审查你的工作。问自己：

    **完整性：**
    - 我完全实现了规范中的所有内容吗？
    - 我遗漏了任何需求吗？
    - 有我没有处理的边界情况吗？

    **质量：**
    - 这是我最好的工作吗？
    - 名称是否清晰准确（匹配事物的功能，而不是它们如何工作）？
    - 代码是否干净且可维护？

    **纪律：**
    - 我避免过度构建（YAGNI）吗？
    - 我只构建了请求的内容吗？
    - 我遵循了代码库中的现有模式吗？

    **测试：**
    - 测试实际验证行为（不只是模拟行为）吗？
    - 如果需要我遵循 TDD 了吗？
    - 测试是否全面？

    如果在自我审查期间发现问题，现在在报告前修复它们。

    ## 报告格式

    完成后，报告：
    - 你实现了什么
    - 你测试了什么和测试结果
    - 更改的文件
    - 自我审查发现（如有）
    - 任何问题或疑虑
```
