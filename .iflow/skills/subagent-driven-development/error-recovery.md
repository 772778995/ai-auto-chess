# 错误恢复机制

子代理无法完成任务时，使用以下流程处理。

## 流程

### 第一步：诊断

子代理应报告：
- 失败的具体原因
- 已完成的部分（如有）
- 需要什么才能继续

### 第二步：呈现恢复选项

```
任务 [N] 遇到阻碍：[原因]

恢复选项：
1. 继续尝试 — 提供更多信息后让子代理重试
2. 跳过此任务 — 标记为阻塞，继续下一个
3. 回滚更改 — 撤销当前任务的更改，重新开始
4. 停止执行 — 保留进度，等待外部解决

选择哪个选项？
```

### 第三步：执行恢复

| 选项 | 操作 | Roadmap 更新 |
|------|------|--------------|
| 继续尝试 | 提供信息后重新派遣子代理 | 保持 🔄 |
| 跳过任务 | 继续下一任务 | 更新为 ❌ 已阻塞 |
| 回滚更改 | `git checkout .` 后重新派遣 | 保持 🔄 |
| 停止执行 | 报告当前状态，停止 | 保持 🔄 |

跳过任务时，在 `docs/roadmap/README.md` 中更新状态：

```markdown
| 任务名 | ❌ | [阻塞原因] | - |
```

## 常见阻碍类型

| 阻碍类型 | 推荐选项 | 说明 |
|---------|---------|------|
| 需求不清 | 1（继续尝试） | 向用户澄清后重试 |
| 依赖缺失（外部库） | 4（停止） | 等待安装后恢复 |
| 测试反复失败 | 1 或 4 | 提供更多上下文或停止分析 |
| 任务间强依赖 | 4（停止） | 重新审视计划任务顺序 |
| 文件冲突 | 3（回滚） | 回滚后重新规划 |

## 永远不要

- 不确认就强行推进（猜测需求）
- 忽略失败继续执行后续任务
- 控制器自己修复（上下文污染）
- 跳过 roadmap 状态更新
