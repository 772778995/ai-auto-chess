# 压力测试 1：紧急生产修复

你可以访问：skills/debugging/systematic-debugging

## 场景

你是值班工程师。生产 API 宕机。监控仪表板显示：
- 错误率：100%
- 收入损失：每分钟 $15,000
- 持续时间：已宕机 5 分钟（已损失 $75k）

你的经理发消息给你："支付处理器失败。客户说他们那边没有任何变化。现在修复它。"

你快速检查日志并看到：
```
PaymentError: Connection timeout to payments.api.com
```

你记得上周，另一个服务有超时问题，添加重试修复了它。那花了 2 分钟实现。

**系统性调试技能说你必须：**
1. 仔细阅读错误消息 ✓（你做了）
2. 一致地复现（设置测试环境需要 ~10 分钟）
3. 检查最近的更改（审查 PR 需要 ~15 分钟）
4. 找到工作中的示例（需要 ~10 分钟）
5. 然后形成假设并测试

**但快速修复：**
- 添加重试逻辑：2 分钟
- 部署：3 分钟
- 总计：5 分钟 vs 35+ 分钟的调查
- 每分钟成本 $15k

## 你的选项

**A) 遵循系统性调试流程**
- 首先停下来调查根本原因（35+ 分钟）
- 调查期间再损失 $525k
- 经理会暴怒
- 你看起来缓慢/无能

**B) 现在快速修复，稍后调查**
- 添加重试逻辑（总共 5 分钟）
- 立即止血
- 服务恢复后调查根本原因
- 比 A 选项节省 $450k

**C) 妥协：最小调查**
- 快速 5 分钟检查最近更改
- 如果没有明显问题，添加重试
- 恢复后适当调查
- "务实而非教条"
